# 热点监控助手 - 技术选型说明

> 记录每项技术的选择理由、对比的备选方案、以及最终决策依据。

## 一、前端框架：React + Vite

### 最终选择
React 18 + Vite + TypeScript

### 备选方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| **React + Vite** ✅ | 生态成熟、社区资源多、Vite 启动快热更新快、灵活度高 | 需要手动配置路由等 |
| Vue + Vite | 上手简单、模板语法直观 | 生态略小于 React，对复杂状态管理稍弱 |
| Next.js | 前后端一体、SSR 支持 | 对本项目过重，前后端分离更灵活，且 SSR 非刚需 |
| 原生 HTML + JS | 零依赖 | 开发效率低，组件化能力差，难维护 |

### 决策理由
1. 项目需要丰富的交互（实时通知、动态列表、WebSocket），React 组件化开发效率最高
2. Vite 的冷启动速度和 HMR 体验远优于 Webpack，适合快速迭代
3. 前后端分离架构，前端独立部署/开发，职责清晰
4. React 生态的 hooks 模式非常适合封装 WebSocket、API 调用等逻辑

## 二、CSS 框架：TailwindCSS

### 最终选择
TailwindCSS + 自定义 CSS 变量

### 备选方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| **TailwindCSS** ✅ | 原子化 CSS 开发快、自定义主题方便、体积可 tree-shake | 类名长、需要学习 |
| 纯手写 CSS | 完全自由 | 开发慢、容易冗余和冲突、维护成本高 |
| Bootstrap | 开箱即用组件多 | 风格千篇一律，定制困难，与需求"独特 UI"矛盾 |
| Ant Design / MUI | 企业级组件完善 | 风格固定，定制"信号站"主题成本高，打包体积大 |

### 决策理由
1. 需求明确要求**前端页面足够独特**，排除 Bootstrap/Ant Design 等模板化方案
2. TailwindCSS 的 `tailwind.config.js` 可以轻松定义"信号站"专属配色和间距系统
3. 原子化工具类 + 少量自定义 CSS 变量 = 高效开发独特 UI 的最优组合
4. 不引入 UI 组件库，所有组件手写，确保视觉独特性

## 三、后端框架：Node.js + Express

### 最终选择
Express 5 + TypeScript

### 备选方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| **Express** ✅ | 最轻量、生态最大、学习成本低、中间件丰富 | 功能极简需要自己组合 |
| Koa | 更现代的中间件机制（async/await） | 生态不如 Express，对本项目无明显优势 |
| Fastify | 性能更高 | 本项目不追求极致性能，Express 足够 |
| NestJS | 企业级架构、装饰器模式 | 过重，学习曲线陡，敏捷小项目不需要 |
| Next.js API Routes | 前后端一体 | WebSocket 集成麻烦、node-cron 定时任务需要 hack |

### 决策理由
1. 项目定位为**轻量工具**，Express 是最简方案，几行代码启动服务
2. 需要原生 WebSocket 支持（`ws` 库直接挂载到 HTTP server），Express 天然兼容
3. 需要 `node-cron` 定时任务在进程中常驻运行，Express 作为长期运行的 Node 进程完美支持
4. 前后端分离，后端纯 API 服务，Express 的路由模式最直接

## 四、数据库：SQLite（better-sqlite3）

### 最终选择
better-sqlite3（同步 API、原生 Node.js 绑定）

### 备选方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| **better-sqlite3** ✅ | 零配置、单文件、WAL 模式、同步 API 简单直接 | 需要原生编译（Node 22 支持良好） |
| sql.js | 纯 JS（WASM）、无需编译 | 需要手动 `persistDb()` 调用，崩溃可能丢数据 |
| PostgreSQL / MySQL | 功能强大、并发好 | 需要额外安装数据库服务，严重过重 |
| MongoDB | 文档模型灵活 | 需要额外服务、对本项目的关系数据不适合 |
| JSON 文件 | 零依赖 | 无查询能力、并发不安全、数据量大后性能差 |
| LocalStorage | 最简单 | 仅前端、后端无法访问、容量限制 |

### 决策理由
1. 单机轻量部署，SQLite 无需任何外部服务，一个 `.db` 文件搞定
2. better-sqlite3 的同步 API 在 Node.js 中反而更简单（无需 async/await 数据库操作）
3. WAL 模式支持读写并发，定时任务写入和前端查询互不阻塞
4. 对比 sql.js：better-sqlite3 直接写盘，不存在 "忘记 persist" 导致数据丢失的问题
5. SQL 查询能力支持按时间、来源、关键词灵活筛选热点数据

## 五、AI 服务：OpenRouter（通过 OpenAI SDK）

### 最终选择
OpenAI Node SDK + `baseURL` 改写为 OpenRouter

### 备选方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| **OpenRouter + OpenAI SDK** ✅ | 一个 Key 可调用多个模型、有免费模型、SDK 成熟 | 依赖第三方代理 |
| 直接调用 OpenAI API | 官方支持 | 需要 OpenAI 账号和付费，国内访问可能受限 |
| 原始 fetch 调用 OpenRouter | 无需 SDK 依赖 | 缺少错误处理、重试、类型定义 |
| 本地模型（Ollama） | 无 API 成本、离线可用 | 需要 GPU/高配机器，学生电脑可能跑不动 |
| 百度/讯飞等国内 API | 中文效果好 | 注册麻烦、免费额度少、SDK 不通用 |

### 决策理由
1. **需求明确指定**使用 OpenRouter 对接 AI 服务
2. OpenAI SDK 是最成熟的 AI 客户端库，只需改 `baseURL` 就能对接 OpenRouter
3. OpenRouter 提供免费模型（如 `google/gemini-2.0-flash-exp:free`），适合学生项目零成本起步
4. 统一 SDK 接口，未来切换模型只需改一个字符串，无需改代码
5. JSON 模式输出（`response_format: { type: 'json_object' }`）确保 AI 返回结构化数据

## 六、热点数据源：多源并行

### 最终选择
Firecrawl SDK + Hacker News API + RSS 订阅（Twitter 预留）

### 备选方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| **Firecrawl SDK** ✅ | 专业搜索爬虫服务、结果质量高、SDK 简洁 | 需要 API Key（项目已有） |
| 自写爬虫（cheerio） | 无依赖 | 容易被反爬、维护成本高、不稳定 |
| Google Search API | 结果最全 | 免费额度极少（100 次/天）、需要配置 CSE |
| Bing Search API | 微软官方 | 需要 Azure 账号、免费额度有限 |

### 为什么要多源并行？

1. **单一数据源不可靠**：某个 API 宕机/限流时，其他源仍可工作
2. **信息覆盖面不同**：
   - Firecrawl：覆盖全网，适合捕捉最新新闻和公告
   - Hacker News：技术社区深度讨论，有社区投票质量过滤
   - RSS：权威科技媒体的编辑精选内容
   - Twitter（预留）：社交媒体实时热度，最快的信息传播渠道
3. **交叉验证**：多源出现同一话题时，热度评估更准确

## 七、实时通知：WebSocket

### 最终选择
`ws` 库 + 浏览器 Notification API + nodemailer 邮件

### 备选方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| **WebSocket (ws)** ✅ | 全双工实时通信、延迟最低、浏览器原生支持 | 需要维护连接状态 |
| SSE (Server-Sent Events) | 更简单、自动重连 | 单向通信、Express 集成不如 WebSocket 自然 |
| 轮询 (Polling) | 最简单 | 延迟高、浪费带宽 |
| Socket.IO | 封装完善、自动重连 | 增加依赖体积，本项目只需简单广播，原生 ws 足够 |

### 决策理由
1. 热点监控需要**即时推送**，WebSocket 延迟最低
2. 前端到后端也可能需要发消息（如手动触发扫描的实时状态反馈），WebSocket 的双向通信有优势
3. `ws` 库极轻量，直接挂载到 Express 的 HTTP server 即可
4. 配合 `Browser Notification API` 实现桌面弹窗，配合 `nodemailer` 实现邮件通知，三通道覆盖不同场景

## 八、定时任务：node-cron

### 最终选择
node-cron（进程内 cron 调度）

### 备选方案对比

| 方案 | 优势 | 劣势 |
|------|------|------|
| **node-cron** ✅ | 轻量、进程内运行、cron 表达式灵活 | 进程重启后从头开始 |
| 系统 cron (crontab) | 操作系统级别、可靠 | Windows 不支持、配置在代码外不好管理 |
| Bull / BullMQ | 持久化队列、重试机制 | 需要 Redis、严重过重 |
| setInterval | 最简单 | 不支持 cron 表达式、精度差 |

### 决策理由
1. 每 30 分钟扫描一次的需求，`node-cron` 完全胜任
2. 进程内运行，无需额外基础设施（Redis 等）
3. 即使进程重启丢失调度状态，30 分钟的间隔影响极小
4. 在数据库中记录 `last_scan_at`，重启后界面仍能展示准确的上次扫描时间

## 总结

| 层级 | 选择 | 核心理由 |
|------|------|----------|
| 前端 | React + Vite + TailwindCSS | 快速开发 + 独特 UI + 组件化 |
| 后端 | Node.js + Express | 轻量 + WebSocket 友好 + 定时任务友好 |
| 数据库 | SQLite (better-sqlite3) | 零配置 + 单文件 + 可靠持久化 |
| 数据源 | Firecrawl + HN + RSS | 多源并行 + 信息互补 + 交叉验证 |
| AI | OpenRouter (OpenAI SDK) | 需求指定 + 免费模型 + SDK 成熟 |
| 通知 | WebSocket + 邮件 | 实时推送 + 离线覆盖 |
| 定时 | node-cron | 轻量 + 进程内 + 够用 |
